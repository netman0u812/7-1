Problem statement

Firewall rule construction at CVS Health is entirely manual, unguided, and disconnected from the underlying network topology.

Engineers build rules by typing zone names freehand, selecting address objects with no validation against topology constraints, and defaulting to Alert-All-SPG because it's the path of least resistance. The result is a 15,629-rule rulebase with no zone coherence enforcement, no systematic security profile coverage, and no formal relationship between the 728 EDL-backed network objects maintained by the CCD platform and the zone names used in Panorama policies.

Three compounding gaps drive this problem:

No NG→zone join record
The CCD platform produces 728 External Dynamic List files organized into Network Groups (NG) and Address Objects (AO). Panorama policies reference zone names like inside, DMZ, clouddmz, WEX-Meritain. No formal mapping between these two namespaces exists anywhere in the estate. Engineers have no validated reference for which zone a given NG/AO belongs in.
No topology-aware rule construction tool
Panorama's native rule editor treats zone names as freetext strings. It does not know that NG-PARTNER-OFFSHORE-CONTRACTOR placed in the trust zone is a hard isolation violation, or that AO-DATACENTER-SCOTTSDALE-AZ-SHEA-DC belongs on the DMZ leg of the hairpin, not the trust leg. No tool enforces these constraints at rule creation time.
No profile selection guidance
Best-practice security profile groups — BP_SPG_DMZ_Model, BP_SPG_User_Internet_Model — are defined in shared on both internet-edge Panoramas but applied to fewer than 2% of rules. Alert-All-SPG (alert-only, no blocking) is the effective default on 15,448 of 15,629 rules.

Security posture gap: The combination of no zone coherence enforcement and near-universal use of Alert-All-SPG means that misconfigured rules are neither detected at creation time nor blocked in the data path. Content inspection infrastructure exists but is inoperative on 98% of traffic.

Design principles

FPMS is built around five principles derived from analysis of the existing estate and the constraints of operating at CVS Health's scale.

Topology-first
The firewall's physical and logical structure — interfaces, zones, zone protection profiles, virtual routers — is the root object. Every policy decision derives from the topology, not the other way around. Zone names are not freetext; they are properties of a specific firewall stack extracted from the live Panorama config.
Named objects only
No raw IP addresses in rules. Every source and destination is a named EDL-backed object (NG, AO, AD group) or a named service object. This is not a style preference — it is the prerequisite for the zone coherence model to function. A raw IP has no zone class; a named NG object does.
Zone coherence as a hard constraint
Each NG and AO category has an expected zone class (TRUST, DMZ, VPN-PARTNER, CLOUD-DMZ). Placing a PARTNER-category object in a TRUST zone is not a warning — it is a hard violation that must be resolved before a rule can be submitted. The mapping is encoded in ng_zone_map.json and enforced by the validation engine.
Profile groups are structural, not decorative
The security profile group applied to a rule is determined by the zone pair, not selected manually. A trust→DMZ rule gets BP_SPG_DMZ_Model. A trust→untrust egress rule gets BP_SPG_User_Internet_Model. A VPN-PARTNER→trust rule gets Trusted-Apps. The rule builder auto-suggests based on the destination zone's default and flags any deviation.
Grounded in live data
All topology data, zone definitions, EDL catalogs, and profile group definitions are extracted directly from the four Panorama config exports using deterministic Python scripts. Nothing is hand-curated or assumed. When the topology changes, the pipeline re-runs and the tool updates.

Active Directory domain controller groups

The AD object set was built from a separate Panorama spreadsheet covering 16 location tags, ~300 individual DC objects, and two aggregate groups. The model uses Panorama's dynamic address group feature: each per-location group (AG_AD-AZ, AG_AD-RI, etc.) has a tag filter expression. Adding a new DC requires only creating the address object with the correct tag — the groups update automatically.

Two traffic patterns consume these groups:

  • AD-to-AD replication: source AG_AD_ALL_FQDN → destination AG_AD_ALL_FQDN, service Active_Directory_Services
  • User-to-DC authentication: source NG-CORP-* (by region) → destination AG_AD-{LOCATION} matching the user's domain

Service repository

The service repo is the one gap not yet closed in the prototype. The planned architecture cross-references three sources:

  • Panorama shared service objects — 6,670 objects already defined, extractable via REST API
  • Applipedia API — GET /restapi/v10.1/Objects/Applications — App-ID name, category, default ports, risk level, application-default flag
  • IANA Service Name Registry — downloadable CSV from iana.org/assignments/service-names-port-numbers, ~10,000 authoritative port-to-name mappings

The cross-reference enables two behaviors in the rule builder: select an App-ID and ports auto-populate from Applipedia defaults; select a port and candidate App-IDs surface. A warning fires when an App-ID is selected with a non-default port (e.g., kerberos on 443), which would require an explicit service object rather than application-default.

The topology model has three structures that share the zone as their common join key. Every policy decision ultimately resolves to a zone name on a specific firewall stack.

Structural flow patterns

Every firewall stack maps to one of four structural patterns. The pattern determines the valid zone combinations, the NAT model, and which NG/AO categories are valid in each position. Building a rule against the wrong pattern produces a coherence violation.

Pattern 1
untrust / outside → extranet DMZ → trust / inside DNAT inbound · SNAT egress
Internet edge hairpin  ·  PAZSHDC-SEC-WIR · RRIWSDC-SEC-EXT-APP · DG_AZ_Egress (5,923 rules) · DG_RI_EXT_APP
Pattern 2
dmz / clouddmz ↔ inside static 1:1 or no NAT
DC DMZ (no internet leg)  ·  mid-dmz · win-dmz · mid-clouddmz · win-clouddmz · mid-qadmz
Pattern 3
trust / inside ↔ outside / internet SNAT egress or 1:1 NAT per server
Simple edge  ·  mid-glr · win-glr · mid-edmz (1,372 NAT) · win-edmz (1,009 NAT) · mid-officeext · HOL-PCI
Pattern 4
VPN-PARTNER ↔ Trust-Inside no NAT — routed IPSec
Partner VPN  ·  LV-3rd_Party (WEX-Meritain · EBIX-Meritain) · mid-officeext (vpn1/2) · Reference model for 3PLZ

The hairpin in Pattern 1 is the most important concept for rule construction. Traffic arriving from the internet hits the untrust zone, is DNAT'd to a server IP in the DMZ zone (the AO-DATACENTER-* object for that server), and then optionally reaches the trust zone (NG-DATACENTER-* for DC backends). The outbound path skips the DMZ entirely: trust → untrust with SNAT to the egress IP pool. No NG or AO object ever appears on the untrust side — it is always any.

A security profile group bundles four distinct profile types into a single named object that attaches to a security rule. The group — not the individual profiles — is what a rule references.

Profile gap is the single largest security posture issue in the estate. The content inspection infrastructure (Threat Prevention, URL Filtering, WildFire) is deployed and licensed but operationally inert on 98% of traffic. FPMS addresses this at rule creation time by auto-assigning the appropriate profile group and flagging any selection of Alert-All-SPG.

Three-structure policy model

The zone protection profile and the security profile group are two different objects serving different purposes that are frequently confused.

Zone protection profile (Structure 2, zone-level)
Attaches to the zone itself, not to individual rules. Runs before rule evaluation. Controls flood protection (SYN, UDP, ICMP), reconnaissance detection (port scan, host sweep), and packet-based attacks. ZPP-no-Spoof is applied to Inside, DMZ-PDE, and Semafone_DMZ on PAZSHDC-SEC-EXT-NET. Missing from outside-facing zones on MID/WIN stacks.
Security profile group (Structure 2, rule-level)
Attaches to individual security rules. Fires after rule match. Controls content inspection — what the firewall does with matched sessions. This is the object that is near-universally set to Alert-All-SPG.
Log forwarding profile (Structure 2, rule-level)
Controls where session logs go. Per-rule. Should send to Panorama for all rules, with selected rules additionally forwarding to SIEM/syslog. Currently inconsistently applied.

Estate analysis findings

HIGH
98% of rules use Alert-All-SPG. 15,448 of 15,629 security rules across DG_AZ_Egress and DG_RI_Egress. Best-practice profiles are defined in shared but applied to fewer than 181 rules combined. No blocking occurs on content inspection across the vast majority of the estate's internet-facing traffic.
HIGH
No NG→zone join record exists. The 728 CCD EDL objects have no formal mapping to zone names in Panorama. Engineers building rules have no validated reference for which zone a given NG/AO category belongs in — the gateway to zone misplacement and unintended policy bypass.
MED
Zone naming is inconsistent across stacks. inside vs Inside (case mismatch), DMZ vs DMZ2 vs DMZ-PDE vs APP_DMZ vs xpe — 22+ distinct zone names across 29 stacks. Rules written with the wrong zone name for the target stack silently fail to match traffic.
MED
eDMZ NAT tables are unmaintained AO equivalents. mid-edmz has 1,372 NAT rules and win-edmz has 1,009 — one per server in AO-DATACENTER-MIDDLETOWN-CT-AETNA-MDC and AO-DATACENTER-WINDSOR-CT-AETNA-WDC respectively. The NAT table and the AO objects are two independently maintained representations of the same server inventory.
MED
Zone protection profiles inconsistently applied. ZPP-no-Spoof applied to inside-facing zones on PAZSHDC-SEC-EXT-NET but not consistently on outside-facing interfaces. MID/WIN outside interfaces carry no zone protection profile.
HIGH
5,161+ rules in DG_AZ_Egress use destination any. Combined with Alert-All-SPG, this means a large portion of the egress rulebase provides neither destination specificity nor content inspection. FPMS addresses this at rule creation time by requiring named destination objects.
INFO
LV-3rd_Party-Stack is the reference implementation for 3PLZ. Named tunnel zones (WEX-Meritain, EBIX-Meritain) per partner, no NAT, MPLS_DMZ for PPOM connectivity. This is the exact PARTNER-VPN pattern 3PLZ should replicate for new partner onboarding.

Rule builder

The rule builder is a topology-aware form that guides engineers through five steps to produce a validated, commit-ready Panorama set command. Every field is constrained by topology data — no freetext zone names, no uncategorized address objects.

Step sequence

StepInputIntelligence
1 — Firewall Panorama, flow pattern filter, stack selector Stack selector populates from full_topology.json. Selecting a stack renders the flow strip showing the structural pattern and NAT model.
2 — Zones Source zone (from), destination zone (to) Zone chips rendered from actual topology — real names, zone class color-coded, connected network displayed. No freetext entry. Selecting a zone updates category hints for step 3.
3 — Objects NG/AO/AD source objects, NG/AO/AD destination objects Category picker filtered by valid categories for the selected zone. Object picker populated from edl_catalog.json. Each added object immediately coherence-checked against zone class — hard violations block submission.
4 — Service + profile Service preset or custom port, profile group Profile group auto-suggested from destination zone's default. Alert-All-SPG selection triggers upgrade warning. Service presets cover AD, LDAP, Kerberos, Web, and any.
5 — Review Rule name, description, log setting Validation summary must show zero hard errors before submit is enabled. Submit triggers review workflow (peer → security team → commit).

Output formats

The rule builder produces three output views in parallel:

  • Palo Alto CLI — complete set commands including EDL registrations, service object definitions, device-group scoping, and profile-setting. Paste-ready for Panorama CLI or commit script.
  • Cisco FMC JSON — equivalent access rule as an FMC REST API payload with ExternalList source/destination objects and zone references.
  • Summary — human-readable rule description with zone classes, subnet counts, NAT model, and profile group — suitable for change ticket description.

API layer

The production FPMS runs a thin FastAPI service over the JSON repos. Four endpoints cover the complete rule lifecycle.

Authentication

API authentication uses CVS SSO via service account tokens. The Panorama API credential used by the pipeline is a read-only service account. The commit path requires a separate write-privileged account gated behind the review workflow approval step.

Commit path

Two commit modes are supported, selecting based on operational maturity and change management requirements.

Mode A — ServiceNow change ticket (default)
POST /commit creates a ServiceNow CHG record with the validated rule payload as a structured attachment. The rule is not pushed to Panorama until the CHG is approved and the engineer manually executes the push. This is the appropriate path for high-risk rules or during initial FPMS rollout.
Mode B — Direct Panorama push (post-approval)
After peer review and security team approval, FPMS calls POST /restapi/v10.1/DeviceGroup/{dg}/PreRulebase/SecurityRules directly, followed by a Panorama commit. The CHG is still created for audit trail but is auto-approved by the system. Appropriate for standard-risk rules once the validation engine has been validated at scale.

3PLZ connection

FPMS is the operational tooling for the 3PLZ (Third-Party Landing Zone) program's partner onboarding workflow. The two programs share the PARTNER-VPN topology pattern and the NG/AO object model as their common foundation.

The LV-3rd_Party-Stack (DG_LV_3rd_Party) is the production reference implementation for 3PLZ's architecture. It demonstrates:

  • One named tunnel zone per partner (WEX-Meritain, EBIX-Meritain) — the zone boundary is the partner isolation policy
  • NG-PARTNER-{NAME} maps 1:1 to the tunnel zone name — the object naming convention enforces the topology
  • No NAT — routed IPSec tunnels with PSK authentication generated by the self-service portal
  • MPLS_DMZ-1/2 for PPOM connectivity alongside IPSec tunnels

When 3PLZ Phase 3D (vendor selection, target October 1, 2026) is complete, FPMS will be the tool used to onboard new partners into the production landing zone. Onboarding a new partner requires: create NG-PARTNER-{NAME} EDL object in CCD platform → pipeline picks it up → FPMS rule builder presents it as a valid source object in the PARTNER-VPN pattern → engineer builds allow rule from NG-PARTNER-{NAME} zone to Trust-Inside zone → submit for review → commit.

The 3PLZ self-service portal generates the PSK per tenant and creates the IPSec tunnel configuration. FPMS handles the downstream firewall policy — the two systems are complementary, not overlapping. The portal owns the tunnel; FPMS owns the rule that permits traffic through it.
