Executive Summary
Scope, program context, and key state findings

CVS Health is replacing McAfee/Skyhigh Secure Web Gateway with Palo Alto NGFW as the web content inspection and DLP enforcement platform for non-Zscaler traffic. Zscaler ZIA serves user-managed endpoints by policy. The Palo Alto estate is designated for machine traffic, server egress, non-standard systems, affiliate connections, and infrastructure — populations that Zscaler does not serve and McAfee/Skyhigh is being retired from.

This assessment documents three things: the current enforcement state of each PA policy architecture; the gap between that state and the Zscaler ZIA reference standard; and the path to delivery parity using PA's native intent-based zone architecture. The framing is state, gap, and path — not a remediation directive.

Key State Findings

CVS PANORAMA

15,603–15,606 security rules (99.5%) use Alert-All-SPG. This profile group overrides all security profile actions to alert/passive at runtime — AV, IPS, WildFire, and URL Filtering are all non-enforcing. Three legacy content lists (Proxy-Zscaler-Blacklist, SOC-Deny-List-Domain, Proxy-WG-General) are structurally present but produce zero blocking action. DLP has a broken object reference. This is the current state for RI One (165 gap subnets), Cumberland (82), Shea DC (36), Distribution, Omnicare, and Corporate locations.

AETNA PANORAMA

The Aetna estate has a significant dependency on proxy.pac-based traffic steering. User internet traffic is currently directed through McAfee/Skyhigh via proxy.pac, not through the PA NGFW. The MID/WIN configurations reflect this operating state — they represent the parallel PA infrastructure during the McAfee transition. This is also the primary reason Zscaler uptake is lower across Aetna spaces: users are steering through proxy.pac to McAfee, not to ZIA. Full PA enforcement deployment requires complete resolution of the proxy.pac dependency.

LAS VEGAS COLO

SPG-LV-Colo-All applies best-practice AV, IPS, file blocking, and WildFire profiles — the most advanced enforcement posture in the PA estate. However only 12 rules are defined, URL filtering applies to only 1 of 12 rules, and key gaps remain (WildFire inline, DNS sinkhole action, credential enforcement, DLP). All Aetna affiliates intended for LV COLO are currently routing through CVS or Aetna Panorama — AD migrated, network routing did not.

PATH TO PARITY

PA's full enforcement capability requires intentional policy design. The intent-based model (named address group → App-ID security rule → dedicated SNAT /32 egress zone → traffic-class-specific profile) delivers functional equivalence to Zscaler's rule chaining. Multiple rules per zone with App-ID context achieves per-application enforcement without requiring full SSL decryption for all controls. PA also provides enforcement capabilities Zscaler cannot — native SSH inspection, L1–L7 network enforcement, east-west visibility — detailed in Section 11.

Network Platforms
PA is not a proxy — L1–L7 architecture requires different policy design

Zscaler ZIA and McAfee/Skyhigh are L7-only proxy platforms — they operate exclusively on HTTP/HTTPS traffic and require proxy.pac or transparent proxy interception. Palo Alto NGFW operates at L1 through L7 — it enforces on all traffic regardless of protocol. A flat permit-any-any-http/https with content filtering on PA produces a poor result because it ignores PA’s enforcement model and discards most of its enforcement surface. The intent-based zone model (Section 11) is the correct approach.

admin.zscalerthree.net — export December 2 2025

Zscaler ZIA represents the enforcement posture defined for user-managed endpoint internet access. The PA estate must deliver equivalent or better controls for its population. Where PA’s L1–L7 architecture provides enforcement beyond Zscaler’s L7 scope, those are noted as PA advantages.

Scope note: Browser isolation (ZIA rules 31, 77, 78, 79 — Isolate action) is a user-tier control for user browser sessions. The PA estate serves machine and server traffic — no user browsers in this population. Browser isolation is correctly delivered by ZIA for user endpoints and is out of scope for this assessment.

June 24 2026 config — HCB estate — proxy.pac-dependent transition state

Proxy.pac Dependency — Transition Context

The Aetna estate’s internet traffic model has a significant dependency on proxy.pac-based traffic steering inherited from the McAfee/Skyhigh deployment. Aetna user internet traffic is currently directed through proxy.pac to McAfee — not through PA. This is the primary driver of lower Zscaler uptake across Aetna locations: proxy.pac intercepts the traffic before it reaches ZIA.

The MID and WIN configurations reflect this operating state — the parallel PA infrastructure during the McAfee transition. The configs are assessed here to establish the transition baseline and identify what the post-migration state needs to look like. A full PA enforcement deployment for the Aetna estate requires:

  • Complete resolution of the proxy.pac dependency — documenting all proxy.pac rules and building PA equivalents before decommission
  • Re-steering Aetna internet traffic from proxy.pac/McAfee to PA (with ZIA handling user-managed endpoints)
  • Building the target PA intent-based zone enforcement policy for the Aetna traffic population
Assessment note: The MID/WIN URL filtering profiles documented below represent the transition baseline, not a completed PA deployment. McAfee/proxy.pac currently carries the enforcement responsibility for this population.

PA vs Zscaler — Architectural Comparison
Why PA requires different policy design than a proxy replacement

The most consequential difference between PA NGFW and Zscaler/McAfee is architecture, not feature parity. Implementing PA as a direct proxy replacement — permit-any-any-http/https with content filtering — produces a poor result because it ignores PA’s enforcement model. The intent-based zone model (Section 11) is the correct approach.

DESIGN PRINCIPLE

A flat security rule (permit source=any destination=any application=http/https profile=Alert-All-SPG) produces no meaningful enforcement regardless of content lists or profiles attached. This is the current state of the CVS estate.

PA App-ID header peeking: PA identifies and enforces policy on Dropbox, Telegram, Mega.nz, Tor, BitTorrent, and 3,000+ other applications without full SSL decryption by inspecting TLS SNI, certificate CN, and HTTP Host headers. This closes the generic-HTTPS problem with significantly lower operational overhead than full decryption for all traffic. Full decryption is only required for payload-level DLP, URL path inspection beyond hostname, and file content inspection for WildFire.

Intent-Based Egress Zone Architecture
How PA delivers Zscaler DLP chaining equivalence for machine and server traffic

The intent-based egress zone model is the structural mechanism by which PA delivers the context-specific enforcement Zscaler achieves through rule chaining. The parallel:

  • Zscaler rule chaining: Rule 1 checks PHI on SharePoint → Rule 2 checks credentials on GitHub → Rule 3 blocks Dropbox for non-approved groups — each rule has independent engines and actions
  • PA intent-based zone model: ESXi-Servers → ESXi-Egress-Zone (SNAT x.x.x.10/32) → profile-esxi [SSH inspect + VMware allowlist + credential DLP] / Windows-Servers → Win-Egress-Zone (SNAT x.x.x.11/32) → profile-windows [AV+IPS+WildFire+PII DLP] / DB-Servers → DB-Egress-Zone (SNAT x.x.x.12/32) → profile-db [bulk-transfer DLP + strict allowlist]

Multiple rules per egress zone with App-ID context deliver per-application enforcement within each traffic class — functionally equivalent to Zscaler’s rule chaining without requiring a proxy architecture.
